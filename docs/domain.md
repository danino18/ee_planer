# Requirements Engine Rules

## Location
`src/domain/generalRequirements/`
- `rulesEngine.ts` — `calculateRequirement(courses, rule)` → progress
- `matchers.ts` — `matchCourse(course, matcher)` predicate logic
- `types.ts` — `GeneralRequirementRule`, `GeneralRequirementProgress`, etc.

## Architecture
Each requirement rule has a `courseMatcher` (discriminated union) and a `targetUnit` (`credits` | `count`). The engine is pure — no side effects, no store access.

## Rules for Changes
- `matchCourse` is pure and must remain pure. No state reads or async logic.
- Requirement type strings: `mandatory`, `elective`, `מלגים`, `sport`, `english`. Used in UI rendering and filtering — do not rename them.
- `targetUnit: 'credits'` uses `course.credits`; `targetUnit: 'count'` uses `1`. This distinction is load-bearing.
- Status values (`completed`, `partial`, `missing`) are used verbatim in CSS class names in components. Do not add new statuses without updating all consumer components.

## Data Source
Track definitions live in `src/data/tracks/*.ts`. General requirement rules are generated by `scripts/sync-general-requirements.mjs` (fetches from Technion UG portal at build time) and written to `src/data/generalRequirements/`.

## Do Not
- Do not hand-edit files in `src/data/generalRequirements/` — they are script-generated. Edit the sync script instead.
- Do not add course IDs or Hebrew names as string literals in `rulesEngine.ts`. All course data belongs in `src/data/`.
