# AGENTS.md

## Stack Snapshot
- Frontend: React 19 + TypeScript + Vite 8
- Styling: Tailwind CSS v4 via `@tailwindcss/vite`
- State: Zustand 5 with localStorage persistence
- Drag and drop: `@dnd-kit`
- Data: SAP course data fetched client-side, Firebase Auth + Firestore sync
- Backend/workers: Firebase Functions in `functions/`
- Package manager: `npm`

## Exact Commands
- Install app deps: `npm ci`
- Install functions deps: `npm --prefix functions ci`
- Start dev app: `npm run dev`
- Build app: `npm run build`
- Typecheck app: `npm run typecheck`
- Lint app: `npm run lint`
- Run tests: `npm run test`
- Analyze bundle / enforce bundle budget: `npm run perf:bundle`
- Build functions: `npm run build:functions`
- Lint functions: `npm run lint:functions`
- Full local validation: `npm run check`

## Important Directories
- `src/components`: planner UI and interaction-heavy screens
- `src/hooks`: derived planner progress and recommendations
- `src/store`: persisted planner state
- `src/services`: SAP/Firebase/network code
- `src/data`: tracks, specializations, generated requirement lists
- `scripts`: repo automation and performance guard scripts
- `functions/src`: Firebase Functions code

## Repo-Specific Standards
- Keep diffs minimal and surgical. Prefer the smallest change that solves the real issue.
- Preserve existing persisted store keys and saved-plan shapes unless migration is explicit.
- UI is RTL/Hebrew. Avoid rewriting user-facing Hebrew copy unless the task requires it.
- `src/data/generalRequirements/generatedCourseLists.ts` is generated by `npm run sync:general-requirements`; update the generator, not just the generated file, when behavior depends on it.
- Tailwind v4 does not scan dynamic interpolated class names safely. Prefer explicit class maps.
- Prefer Zustand selectors over broad `usePlanStore()` subscriptions in hot components.
- Memoize or defer only when a concrete hot path justifies it.

## Performance Rules
- Optimize first-load JS before adding new dependencies.
- Lazy-load non-critical UI such as heavy modals and detail panels when safe.
- Keep search/filtering off the hottest render path with memoized indexes and deferred query handling where useful.
- Avoid broad store subscriptions in `App`, planner grid, course search, and course cards.
- Treat bundle size regressions as failures if `npm run perf:bundle` fails.

## Security Rules
- Never commit secrets from `.env.local`; keep example/env docs sanitized.
- Validate remote fetches and handle failed responses explicitly.
- Keep Firestore writes sanitized; strip `undefined` values before persistence.
- Do not introduce unsafe HTML rendering or `dangerouslySetInnerHTML` without a compelling reason and sanitization.
- Prefer least-privilege client behavior; do not expose internal-only data in the UI just for convenience.

## Definition Of Done
- Relevant code is implemented in the repo, not just described.
- `npm run lint` passes.
- `npm run test` passes.
- `npm run build` passes.
- `npm run build:functions` and `npm run lint:functions` pass when functions are affected or as part of full validation.
- Performance-sensitive changes are validated with measurable evidence such as bundle output or explicit hot-path render reductions.
- Docs and workflow instructions stay aligned with the actual repo commands.

## Preferred Workflow
1. Inspect the real code path first.
2. Measure a baseline before performance claims.
3. Implement the highest-impact low-risk improvements first.
4. Re-run the smallest relevant validation set after each meaningful batch.
5. Summarize exact files changed, commands run, and anything not verified.
